# Benchmark: OpenAI native compaction vs. token-budget-matched textual summary

## Question

Does OpenAI's opaque Responses compaction preserve useful conversation state better than a conventional textual summary when both are produced by the same model and the textual summary receives the native compaction pass's reported output-token budget?

This benchmark does **not** attempt to decrypt OpenAI's artifact or prove what representation it contains. It measures downstream behavior.

## Compared conditions

Each synthetic conversation is evaluated under three conditions:

1. **Full context control** — original history plus a shared recent tail. This establishes whether the model can answer the fixture when no compression is used.
2. **Native compaction** — one opaque `compaction` item produced by Responses compaction v2, plus the same shared recent tail.
3. **Text summary** — a textual continuation summary generated by the same model, plus the same shared recent tail.

For each paired trial, the text request's `max_output_tokens` equals `output_tokens` reported by the native compaction request. This is the best externally observable budget match available; the API does not expose the decrypted artifact's internal token length.

The native and textual compactors do not see the later benchmark questions.

## What is tested

Every fixture contains 75 questions, fifteen in each category:

- **Exact recall:** codes, ports, paths, checksums, and names.
- **Relational state:** dependency edges, deployment location, reporting relationships, and key location.
- **Tool history:** exact values from read, test, edit, database, and deployment tool results.
- **Distractor resolution:** final authoritative values after explicit corrections, surrounded by many unrelated decoys.
- **Task continuation:** done, in-progress, and blocked work; blockers; and the immediate next action.

Fixtures contain hundreds of messages and deterministic unrelated archival distractors. Answers are synthetic and fixture-specific. Identifier payloads are derived from SHA-256 so omitted values cannot be reconstructed by extrapolating a visible arithmetic sequence.

## Controls

- Same OpenAI model for native compaction, text summarization, and evaluation.
- Native and text arms receive the same shared recent tail.
- Same evaluation instructions, tools, response schema, and reasoning level.
- Strict JSON-schema output and deterministic exact-string scoring; no LLM judge.
- Evaluation-arm order alternates by trial.
- `store: false` on ordinary Responses requests.
- Encrypted artifact contents are never written to disk. Records retain only SHA-256, byte length, usage, latency, summaries, answers, and scores.

## Reproduction

Requirements:

- Node 22+
- `OPENAI_API_KEY`
- project dependencies installed

Run local checks:

```bash
node --experimental-strip-types benchmarks/native-vs-text/self-test.ts
npx tsc -p benchmarks/native-vs-text/tsconfig.json
```

Pilot:

```bash
node --experimental-strip-types benchmarks/native-vs-text/run.ts \
  --model gpt-5.6-sol --fixtures 1 --trials 1
```

Full run:

```bash
node --experimental-strip-types benchmarks/native-vs-text/run.ts \
  --model gpt-5.6-sol --fixtures 6 --trials 2
```

Analyze a run:

```bash
node --experimental-strip-types benchmarks/native-vs-text/analyze.ts \
  benchmarks/native-vs-text/results/<run-directory>
```

## Records

Each run directory contains:

- `manifest.json` — protocol, model, dimensions, and scoring rules.
- `fixtures.json` — complete generated conversations, questions, and expected answers.
- `trials.jsonl` — per-trial summaries, response IDs, usage, latency, actual answers, and exact scores. Opaque ciphertext is represented only by hash and byte length.
- `scores.csv` — flattened evidence suitable for independent analysis.
- `summary.json` — aggregate/category/fixture scores, paired disagreements, usage, and costs.
- `GENERATED_RESULTS.md` — machine-generated tables.

Run the stronger text-only sensitivity against an existing reference run:

```bash
node --experimental-strip-types benchmarks/native-vs-text/run-dense-text-variant.ts \
  benchmarks/native-vs-text/final-results/<run-directory>
```

See `REPORT.md` for the interpretation of the retained reference run.

## Important limitations

- Matching `max_output_tokens` to native `output_tokens` is not proof of equal effective information capacity.
- The same model generates and consumes each representation, so results may not transfer to other models.
- Synthetic exact-answer tasks measure state preservation, not all aspects of real software-engineering continuation.
- Repeated trials over the same fixture are not independent samples. Statistical intervals over question observations are descriptive; fixture-level consistency and paired disagreements should also be inspected.
- A native advantage would show better behavioral preservation, not by itself prove a latent-space representation. An optimized hidden textual or structured summary could produce the same result.
